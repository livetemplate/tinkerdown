# LLM-Generated Tinkerdown Apps

> ⚠️ **NOTE**: These are research examples demonstrating LLM-generation patterns.
> Examples using `exec` sources run shell commands - review generated code before running.
> These examples may not work with the current tinkerdown version.

This directory contains examples of apps that can be reliably generated by LLMs (Claude, GPT-4, etc.).

## The Pitch

```
Traditional:  Prompt → LLM → [React + API + DB code] → npm install → build → deploy → works
Tinkerdown:   Prompt → LLM → [Single markdown file] → tinkerdown serve → works
```

## Why Markdown is More Deterministic for LLMs

| Factor | React/JS | Tinkerdown Markdown |
|--------|----------|---------------------|
| Syntax complexity | High (JSX, hooks, imports) | Low (YAML + HTML) |
| Ways to fail | Many (build errors, deps) | Few (just serve it) |
| Output size | 10+ files | 1 file |
| Iteration loop | Change → build → check | Change → auto-reload |

## Prompt Templates

### Template 1: Personal Tracker

**Prompt:**
```
Create a tinkerdown app to track [THING].
I want to add new [THINGS], see them in a list, and delete them.
Store the data in the markdown file itself.
```

**What LLM generates:** Self-contained CRUD app with markdown data source.

**Example:** `01-job-tracker/`

---

### Template 2: Data Viewer

**Prompt:**
```
Create a tinkerdown app that shows data from [SOURCE].
Display it in a table with columns [COLUMNS].
```

**What LLM generates:** Read-only dashboard with auto-rendering table.

**Example:** `02-api-dashboard/`

---

### Template 3: System Status

**Prompt:**
```
Create a tinkerdown app that shows the output of these commands:
- [COMMAND 1]
- [COMMAND 2]
Display each in a table.
```

**What LLM generates:** Exec-based status page.

**Example:** `03-system-status/`

---

### Template 4: Form Collector

**Prompt:**
```
Create a tinkerdown app with a form to collect [FIELDS].
Save submissions to a SQLite database.
Show all submissions in a table below the form.
```

**What LLM generates:** Form + SQLite CRUD app.

**Example:** `04-feedback-form/`

---

### Template 5: Active Meeting Notes

**Prompt:**
```
Create a tinkerdown meeting notes template.
Include sections for discussion notes (regular markdown),
action items (interactive list I can add to),
and decisions (another interactive list).
Store data in the markdown file.
```

**What LLM generates:** Hybrid document with static notes + interactive sections.

**Example:** `06-meeting-notes/`

---

### Template 6: Personal Knowledge Tracker

**Prompt:**
```
Create a tinkerdown app to track [ITEMS] I want to [DO].
Include stages: [STAGE 1], [STAGE 2], [STAGE 3].
Each stage has different fields.
Store everything in the markdown file.
```

**What LLM generates:** Multi-stage workflow tracker.

**Example:** `07-reading-list/`

## LLM Reference Card

When prompting an LLM, include this reference:

```
# Tinkerdown Quick Reference

## Structure
- Frontmatter (YAML) defines sources
- Body (markdown + HTML) defines UI
- Code blocks with ```lvt``` for interactive sections

## Data Sources
sources:
  name:
    type: json|csv|rest|pg|sqlite|exec|markdown
    file: path.json          # for json/csv
    url: https://...         # for rest
    query: SELECT * FROM x   # for pg/sqlite
    cmd: ./script.sh         # for exec
    anchor: "#section"       # for markdown

## Auto-Rendering Components
<table lvt-source="name" lvt-columns="col1,col2" lvt-empty="No data">
</table>

<ul lvt-source="name" lvt-field="title">
</ul>

<select lvt-source="name" lvt-value="id" lvt-label="name">
</select>

## Actions
<form lvt-submit="action_name">
  <input name="field">
  <button type="submit">Submit</button>
</form>

<button lvt-click="action_name" lvt-data-id="123">Click</button>

## Table with Actions
<table lvt-source="name" lvt-columns="col1,col2" lvt-actions="edit:Edit,delete:Delete">
</table>
```

## Success Metrics

An LLM-generated app is successful if:

1. **Runs on first try** - `tinkerdown serve` works without edits
2. **Data displays** - Tables/lists show data from sources
3. **Actions work** - Forms submit, buttons trigger actions
4. **Self-documenting** - Markdown explains what the app does

---

## The "Active Notes" Concept

Tinkerdown's killer feature: **notes that do things**.

| Static Markdown | Tinkerdown "Active Notes" |
|-----------------|---------------------------|
| `- [ ] Do thing` (manual toggle) | Interactive checkboxes |
| Copy-paste to add items | Form to add items |
| Read-only document | Document + working app |
| Separate tools for tracking | Tracking built in |
| "I'll organize this later" | Already organized |

This is why **personal productivity + note-taking** is the ideal use case:
- Markdown is already how people take notes
- LLMs are great at generating markdown
- The file stays readable/editable by hand
- Version control just works
